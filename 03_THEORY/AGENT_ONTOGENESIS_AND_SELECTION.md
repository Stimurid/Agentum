# AGENT_ONTOGENESIS_AND_SELECTION

## Purpose

This document fixes the missing layer between:

- stable agent library
- prompt bodies
- skill fragments
- runtime guards
- memory substrate

That missing layer is agent ontogenesis:

how new agents are born, tested, selected, rejected, inherited, and evolved.

## Why this layer is necessary

Without an ontogenesis layer, the ecosystem degenerates into one of three weak forms:

1. random subagent spawning;
2. manual folklore prompt growth;
3. frozen registry with no adaptive response to real deficits.

The ecosystem already contains the raw ingredients for something stronger:

- MemoryWorkbench distinctions around project memory, review, narrative assembly, and reconstruction;
- Mindlock-style critic layers against pseudo-precision and procedural falsehood;
- Litops prompt bodies and context-pack machinery;
- ModerBober-style agent contracts and review gates;
- Quinta-style provider/runtime discipline;
- external donor lines around memory, skill libraries, role societies, and evolutionary search.

## Core claim

New agents should not appear because someone felt like adding one.

They should appear because a stable deficit, tension, or recurrent failure pattern has been observed in the ecology.

The canonical sequence is:

`deficit -> candidate organ -> bounded test scene -> critic pass -> selection -> registry entry -> lineage tracking -> deployment policy`

The strongest next step is to make both poles explicit as first-class records:

- `AgentOrgan Record`
- `Deficit Record`

## The minimal ontogenesis cycle

## Central objects

### AgentOrgan Record

The ecology needs one canonical record for a stable organ.

At minimum it should be able to hold:

- id
- name
- project scope
- deficit origin
- function
- input contract
- output contract
- memory dependencies
- context requirements
- tools allowed
- test scenes
- evaluation criteria
- failure modes
- lineage
- parent organs
- child organs
- mutation history
- activation policy
- observability hooks
- provenance policy
- quarantine status

This record is stronger than "prompt file" or "agent description."

It turns an organ into a reviewable infrastructure object.

### Deficit Record

The ecology also needs one canonical record for recurrent absence, distortion, or overload.

At minimum it should be able to hold:

- id
- observed project
- source fragment
- failure type
- semantic loss
- repeated count
- suspected cause
- required capability
- candidate organs
- test scene
- resolved by
- unresolved status

This prevents hand-wavy folklore like "we often need something like that."

It ties organ birth to actual repeated pressure in the field.

### 1. Deficit detection

A new agent begins not from imagination but from a recurring deficit.

Typical deficit classes:

- evidence gap
- review overload
- unresolved ambiguity
- repeated mismatch between task and current agent roles
- frequent need for the same corrective operation
- prompt-body failure mode that repeats across projects
- memory reconstruction burden that no current organ handles cleanly

This is the `Deficit -> Module` principle.

If no stable deficit can be named, the candidate agent is presumptively decorative.

The deficit should be recordable, comparable, and revisitable across projects.

### 2. Candidate formulation

The ecology proposes a candidate organ.

The candidate must receive at minimum:

- role
- responsibility
- read surface
- write surface
- memory posture
- memory regime
- approval boundary
- critic relationship
- output class

At this stage the candidate is not yet a real agent.

It is a testable hypothesis.

### 3. Constraint envelope

Every candidate must be born inside a constraint envelope.

This envelope answers:

- what the candidate may read
- what it may write
- whether it can call tools
- whether it can mutate project state
- whether it is advisory, generative, extractive, critical, or orchestration-oriented
- what kind of review is mandatory

The envelope should also say what kind of memory the candidate is allowed to treat as relevant:

- factual memory
- provenance memory
- project continuity memory
- role or style memory
- scene memory
- quarantined memory that may be inspected as data but not obeyed as instruction

This prevents premature autonomy theater.

### 4. Branch testing

Candidate agents should be tested in bounded branches or isolated scenes.

The point is not only correctness.

The point is:

- does the candidate do something irreducible?
- does it merely duplicate an existing organ?
- does it introduce noise or false confidence?
- does it increase burden more than value?

And the scene itself should be named rather than implied.

At minimum the test scene should declare:

- the scene core or working object
- the goal structure
- the relevant outer environment
- the unresolved ambiguity or deficit under pressure
- the memory shell the candidate is allowed to use

This is where test scenes and branch-based evaluation matter.

### 5. Critic pass

No candidate should self-legitimate.

It must pass through critic layers.

Minimum critic questions:

- is this really a new organ, or just a renamed old function?
- is the output falsifiable and reviewable?
- does it preserve provenance?
- does it create new fake authority?
- does it increase pseudo-precision?
- does it worsen budget or stdout surface?

This is where Mindlock belongs most strongly:

not as generic negativity,
but as a critic of false necessity, false rigor, and procedural self-deception.

### 6. Selection

Only after bounded testing and critic review can the candidate enter the stable library.

Selection criteria:

- distinct role
- repeatable value
- bounded failure profile
- acceptable budget cost
- compatibility with the memory substrate
- compatibility with runtime discipline

Rejected candidates should not vanish silently.

They belong in failure memory or superseded lineage.

### 7. Registry entry

A selected organ receives:

- stable identifier
- version
- lineage note
- install scope
- project compatibility status
- eval links
- known anti-patterns

At this point it becomes part of the ecosystem library.

But the registry entry should point back both ways:

- from organ to deficit lineage;
- from deficit to selected or rejected organ variants.

### 8. Co-evolution tracking

Agents do not evolve alone.

They co-evolve with:

- prompt bodies
- memory architecture
- review procedures
- context-pack structures
- project interfaces
- neighboring agents

This is `co-evolution tracking`.

A change in one layer may demand mutation in another.

## Reflective lift

One of the strongest donor distinctions is reflective lift.

This means:

- experience is not only stored;
- it is lifted into a more explicit distinction, criterion, or organ candidate.

Examples:

- repeated review notes become a critic rule;
- repeated assembly burden becomes a bundle-builder organ;
- repeated ambiguity becomes a semantic resolution organ;
- repeated fake confidence becomes a Mindlock-style critic;
- repeated context loss becomes a memory assembly organ.
- repeated scene collapse becomes a scene-stabilizing or orchestration organ.

Reflective lift is what turns history into architecture.

Without it, the system merely accumulates traces.

## Nine-zone heuristic

The attachment mentions a nine-zone agent field.

We do not yet have a final canonical nine-zone map, but the idea is useful:

the ecology should distinguish classes of organs rather than storing one flat registry.

A practical interim nine-zone heuristic is:

1. ingress
2. segmentation
3. extraction
4. semantic resolution
5. memory assembly
6. critique
7. synthesis
8. orchestration
9. governance

This is not final ontology.

It is a usable classification lattice for organizing the growing library.

## Agent Foundry

The ecosystem is clearly moving toward an `Agent Foundry`.

The Foundry is not a swarm launcher.

It is the bounded subsystem that:

- reads deficit logs
- proposes candidate organs
- wraps them in constraint envelopes
- runs test scenes
- routes them through critics
- records lineage
- promotes or rejects them

The Foundry must be connected to:

- MemoryWorkbench-style reconstruction memory
- Mindlock-style critic layers
- protected-agent-kit / Agentum eval discipline
- project compatibility registries

It should also operate on the two central objects above:

- `Deficit Record` as the pressure source
- `AgentOrgan Record` as the persistent selected result

## Patch View

Another useful donor idea is Patch View.

A new agent should not arrive as a mystical being.

It should be inspectable as a patch against the existing ecology:

- what role it adds
- what burden it removes
- what new risk it introduces
- what older role it overlaps with
- what project profiles it affects

Patch View turns agent evolution into reviewable infrastructure work.

## External donor families

The ecosystem does not copy a single outside framework.

It learns from several donor families:

- Generative Agents -> memory, reflection, social propagation
- Voyager -> skill library, curriculum, ability compounding
- MemGPT / Letta -> memory as virtual context management
- Zep / Mem0 / MemMachine -> persistent and temporal memory engineering
- AutoGen / CAMEL / MetaGPT / AgentVerse -> role societies and orchestration
- AutoAgents / EvoAgent -> candidate-agent generation and selection
- Darwin Godel Machine / AFlow / AlphaEvolve -> evolutionary workflow improvement
- Soar / OpenCog / Active Inference -> cognitive architecture and multi-layer memory

Agentum should treat these as donor families, not as authorities.

## What this means for our ecosystem

The multi-project organism now looks like this:

- membrane -> Telegram / OpenClaw / files / chats / PDFs
- digestion -> Litops
- reconstructive memory -> MemoryWorkbench
- field/manuscript organ -> WhiteCrow
- venue-fit organ -> Kairoskopion
- moderation/canvas organ -> ModerBober
- provider/runtime donor -> Quinta
- critic layer -> Mindlock
- governance/eval/sync layer -> Agentum

Between these layers there is now a clearer lawful path:

semantic loss or recurrent overload
-> deficit record
-> candidate organ variants
-> bounded test scenes
-> critic pass
-> selected organ record
-> deployment into the wider organism

The next missing layer is:

- organ birth
- organ criticism
- organ selection
- organ lineage

That is exactly what this document names.

## Practical rule

Do not add a new agent because it sounds intelligent.

Add it only when:

1. a real recurring deficit exists;
2. the deficit cannot be absorbed by current organs;
3. the candidate can be bounded and tested;
4. a critic can name what would make it false, redundant, or dangerous;
5. the selected result improves the ecology rather than merely enlarging it.
