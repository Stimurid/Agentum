# AGENT_FOUNDRY_PIPELINE

## Purpose

This document defines the minimal lawful pipeline for organ birth, testing, selection, and retirement in the Agentum ecosystem.

The goal is not to produce an "agent factory."

The goal is to prevent two opposite failures:

- random proliferation of decorative agents;
- repeated pressure that never hardens into a durable organ.

The Foundry is the bounded evolution layer between:

- `Deficit Record`
- `AgentOrgan Record`
- test scenes
- critic pass
- registry promotion
- install policy

## Core claim

The ecosystem should not answer repeated pain with folklore.

It should answer it with a lawful pipeline:

`pressure -> deficit record -> candidate set -> bounded scene tests -> critic pass -> selection -> organ registry -> eval binding -> install policy`

This is the smallest viable evolution loop for a multi-project cognitive organism.

## What the Foundry is not

The Foundry is not:

- an always-on subagent swarm;
- autonomous self-modification without review;
- benchmark gaming for its own sake;
- "generate 100 agents and keep the smartest";
- a decorative ontology wrapper around normal prompt edits.

It is a reviewable pipeline that converts recurrent project pressure into bounded architecture change.

## Entry conditions

The Foundry should start only when at least one of these is true:

- the same deficit recurs across tasks;
- the same manual workaround is repeated;
- the same false completion or routing failure keeps returning;
- current organs overlap but still leave a stable gap;
- a scene repeatedly collapses in the same place;
- a critic identifies pseudo-rigor caused by a missing organ or missing boundary.

If none of these are true, do not start the Foundry.

Use a smaller fix instead.

## Pipeline stages

### 1. Pressure capture

Input:

- run traces
- human review notes
- eval failures
- registry inconsistencies
- memory reconstruction reports

Output:

- compact pressure note or direct candidate `Deficit Record`

Questions:

- what actually failed?
- what semantic or operational damage occurred?
- is this recurrent or one-off?

### 2. Deficit normalization

Input:

- raw pressure signal

Output:

- stable `Deficit Record`

Required normalization:

- separate symptom from cause;
- name semantic loss explicitly;
- name scene signature;
- name memory signature if present;
- consider non-organ fixes before proposing a new organ.

Gate:

- if the deficit cannot be stated clearly, it is not ready for organ birth.

### 3. Candidate set generation

Input:

- normalized deficit

Output:

- bounded candidate set

Allowed outputs:

- one to three organ candidates;
- one or more non-organ fixes;
- explicit "no new organ needed" result.

Rules:

- generate variants, not only one sacred answer;
- keep variants small and role-distinct;
- do not let one candidate pretend to solve the whole organism.

### 4. Constraint envelope

Every candidate must receive a constraint envelope before scene testing.

Minimum fields:

- project scope
- working object
- memory regimes used
- memory authority limits
- tools allowed
- write surface
- approval boundary
- expected runtime risks

If this envelope is missing, the candidate is still fantasy.

### 5. Bounded scene tests

Input:

- candidate set with envelopes

Output:

- scene-tested candidate evidence

Each candidate should be tested against named scenes that make the deficit visible.

A scene should declare:

- scene core
- goal structure
- outer environment
- relevant memory shell
- expected failure pressure
- success criterion

Good scene classes:

- foreign prompt ingress
- long continuity recovery
- source-grounded rewrite
- critic pass under pseudo-rigor pressure
- mixed dynamic pressure with memory, stdout, and project bleed

### 6. Critic pass

No candidate self-legitimates.

Each candidate passes through critic questions:

- is this a real organ or a renamed old function?
- does it reduce the deficit or only rename it?
- what would falsify its usefulness?
- does it worsen cost, stdout, or complexity surface?
- does it blur data/instruction boundaries?
- does it create monoculture or fake safety theater?

The critic may return:

- reject
- revise
- compare with sibling candidate
- promote to controlled selection

### 7. Selection

Selection is not "which idea sounds coolest."

Selection criteria:

- real deficit match
- scene performance
- bounded failure profile
- acceptable runtime risk
- compatibility with memory architecture
- compatibility with governance and install boundaries

Possible outcomes:

- select one candidate
- merge two candidates
- keep candidate quarantined
- choose non-organ fix instead
- reject entire candidate set

### 8. Registry promotion

Selected result receives:

- `AgentOrgan Record`
- organ registry entry
- linked deficit reference
- failure mode links
- eval links
- install scope

Rejected candidates should still leave lineage traces in notes or mutation history.

Silent forgetting is a regression.

### 9. Eval binding

No organ becomes stable without eval attachment.

Each promoted organ should answer:

- what deficits it addresses;
- what eval scenarios test those deficits;
- what dynamic seeds stress its boundaries;
- what failure ids remain active despite it.

This is where the Foundry connects to `06_EVALS`.

### 10. Install policy

No organ is universal by default.

After eval binding, decide:

- ecosystem-only
- reusable in reviewed profiles
- target-project-specific
- quarantined experimental

Install policy must answer:

- where the organ may go
- where it must not go
- whether it belongs in runtime, docs, evals, or governance only

### 11. Co-evolution review

Promotion is not the end.

Each selected organ may force updates in:

- deficit registry
- eval matrix
- install profiles
- handoff prompts
- memory architecture
- neighboring organs

This is the co-evolution loop.

## Central objects in the Foundry

The Foundry should operate through these objects, not through vibes:

- `Deficit Record`
- `AgentOrgan Record`
- test scene
- critic note
- eval binding
- install scope

If one of these is missing, promotion should usually stop.

## Minimal review cadence

The smallest viable cadence is:

1. observe recurrent pressure
2. write or update deficit record
3. propose bounded candidate set
4. run scene tests
5. run critic pass
6. select / reject / quarantine
7. update registries and eval links

This can be done manually in v1.

Automation may come later.

## Failure patterns inside the Foundry

Watch for these:

- deficit inflation from one dramatic event;
- candidate monoculture;
- critic theater with no real invalidation path;
- scene tests too easy to matter;
- promotion without eval binding;
- promotion without install boundary;
- organ proliferation with no retirement path;
- permanent quarantine that never resolves into decision.

## Practical rule

A new organ is not justified because the ecosystem is intellectually ambitious.

It is justified only when:

- recurrent pressure is visible;
- a stable deficit can be named;
- a bounded candidate can be scene-tested;
- a critic can attack it honestly;
- the selected result improves the organism more than it enlarges it.

That is the law of the Foundry.
